# CLAUDE.md — git-sports-guntar

## Project Overview

Automated fitness dashboard that syncs Garmin/Strava activities into GitHub-style heatmaps, deployed as a static GitHub Pages site. Fork of `aspain/git-sweaty`.

- **Live site**: https://guntarion.github.io/git-sports-guntar/
- **Source**: Garmin Connect (primary)
- **Stack**: Python 3.11 ETL pipeline + vanilla JS frontend (no framework)
- **Deploy**: GitHub Actions → GitHub Pages

## Commands

```bash
# Run full pipeline locally (requires config.local.yaml)
cd scripts && python run_pipeline.py

# Run tests
cd scripts && python -m pytest ../tests/ -v

# Trigger sync (GitHub CLI)
gh workflow run "Sync Heatmaps" --field source=garmin
gh workflow run "Sync Heatmaps" --field source=garmin --field full_backfill=true

# Local dev server
bash scripts/dev_dashboard.sh
```

## Architecture

### Data Pipeline (run_pipeline.py orchestrates all)

```
Garmin/Strava API
  → sync_garmin.py / sync_strava.py    (activities/raw/{source}/*.json)
  → enrich_garmin.py                    (activities/enriched/garmin/*.json — HR zones, splits)
  → normalize.py                        (data/activities_normalized.json)
  → aggregate.py                        (data/daily_aggregates.json)
  → generate_heatmaps.py               (site/data.json — heatmap + stats)
  → generate_activities.py             (site/activities.json — per-activity detail)
  → generate_ai_insights.py            (site/ai_insights.json — Qwen AI coaching)
  → sync_db.py                          (PostgreSQL — optional)
```

### Frontend Pages (all self-contained HTML with inline CSS/JS)

| Page | File | Data Source | Purpose |
|------|------|-------------|---------|
| Dashboard | `site/index.html` + `site/app.js` | `data.json` | Heatmap grid, type/year filters |
| Activities | `site/activities.html` | `activities.json` | Per-activity detail with HR zones & splits |
| Analytics | `site/analytics.html` | `activities.json` + `ai_insights.json` | Charts, monthly progress, AI Coach panel |
| Records | `site/records.html` | `activities.json` | Running leaderboards, split rankings, PRs |

### Branching Strategy

- `main` — source code, site HTML/CSS/JS, workflows
- `dashboard-data` — persisted pipeline state (data/, site/*.json, activities/enriched/)
- Pages deployed from `site/` directory after merging data from `dashboard-data`

## Key Files

## Environment Variables (GitHub Secrets/Variables)

**Required (Garmin source):**
- `GARMIN_TOKENS_B64` — Base64 OAuth token store, OR
- `GARMIN_EMAIL` + `GARMIN_PASSWORD` — fallback auth

**Optional:**
- `QWEN_API_KEY` — Qwen AI for coaching insights (DashScope Intl)
- `QWEN_ENDPOINT` — Override endpoint (default: `dashscope-intl.aliyuncs.com`)
- `DATABASE_URL` — PostgreSQL connection for sync_db.py
- `DASHBOARD_SOURCE` — "garmin" or "strava" (var)

## Data Filtering Rules

These filters are applied consistently across frontend and backend:
- **Exclude runs < 500m** — too short to be meaningful
- **Exclude pace > 15:00/km (900s)** — anomalous data (treadmill glitches)
- **Exclude splits < 500m** — partial km at end of run (records.html only)

## Code Style & Patterns

- Python scripts: no framework, stdlib + minimal deps (`requests`, `garminconnect`, `psycopg2`)
- Frontend: vanilla JS, no build step, inline in HTML files
- Config: YAML with `config.yaml` (base) + `config.local.yaml` (secrets, gitignored)
- JSON I/O: atomic writes via `.tmp` + rename (`utils.write_json`)
- All dates: ISO 8601 (`YYYY-MM-DD` for display, full ISO for timestamps)

## Gotchas

- `config.local.yaml` is generated by workflow from secrets — never commit it
- `dashboard-data` branch is auto-managed; don't edit manually
- `site/app.js` is 186KB — the main heatmap SPA, complex touch handling
- Garmin API has strict rate limits; `sync_garmin.py` handles backoff
- AI insights are non-fatal — pipeline succeeds even if Qwen API fails
- Split data comes from enrichment; only Garmin source has splits/HR zones
- The `__APP_VERSION__` placeholder in index.html is replaced at deploy time
