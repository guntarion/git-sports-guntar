---
PATH A: Setup Strava

A1. Buat Strava API App

1. Buka https://www.strava.com/settings/api
2. Buat aplikasi baru, isi:
 - Application Name: bebas (misal git-sweaty)
 - Category: Other
 - Authorization Callback Domain: localhost
3. Catat Client ID dan Client Secret

A2. Dapatkan Refresh Token

Buka URL ini di browser (ganti YOUR_CLIENT_ID):
https://www.strava.com/oauth/authorize?client_id=YOUR_CLIENT_ID&response_type=code&redirect_uri=http%3A%2F%2Flocalhost%2Fexchange_token&ap
proval_prompt=force&scope=read%2Cactivity%3Aread_all
- Klik Authorize
- Kamu akan redirect ke localhost yang error (normal) — salin nilai code dari URL

A3. Tukar Code → Refresh Token

curl -sS -X POST https://www.strava.com/oauth/token \
 -d client_id=YOUR_CLIENT_ID \
 -d client_secret=YOUR_CLIENT_SECRET \
 -d code=YOUR_CODE \
 -d grant_type=authorization_code
Dari response JSON, salin nilai refresh_token.

A4. Set GitHub Secrets

gh secret set STRAVA_CLIENT_ID --body "YOUR_CLIENT_ID" --repo guntarion/git-sports-guntar
gh secret set STRAVA_CLIENT_SECRET --body "YOUR_CLIENT_SECRET" --repo guntarion/git-sports-guntar
gh secret set STRAVA_REFRESH_TOKEN --body "YOUR_REFRESH_TOKEN" --repo guntarion/git-sports-guntar

    localStorage vs PostgreSQL

  localStorage adalah storage bawaan browser. Data disimpan di device/browser pengguna, bukan di server. File storage.js menyediakan operasi
   CRUD (Create/Read/Update/Delete) yang membaca dan menulis ke localStorage — jadi Journal dan Todo kamu hidup hanya di browser kamu.

  Tidak ada hubungan dengan PostgreSQL yang sudah ada di pipeline. PostgreSQL (via sync_db.py) menyimpan data aktivitas dari Garmin — itu
  backend pipeline. Tapi Journal/Todo tidak pernah menyentuh database itu.

  Kenapa localStorage, bukan database?

  GitHub Pages = static site. Artinya:
  - Hanya bisa serve file HTML/CSS/JS — tidak ada server-side code
  - Tidak bisa menjalankan Python, Node.js, PHP, atau API endpoint
  - Tidak bisa menerima POST request atau menyimpan data dari user

  Jadi satu-satunya tempat menyimpan data dari interaksi user di browser adalah localStorage.

  Keterbatasan localStorage (in the long run)

  ┌─────────────────┬────────────────────────────────────────────────────┐
  │  Keterbatasan   │                       Dampak                       │
  ├─────────────────┼────────────────────────────────────────────────────┤
  │ Device-bound    │ Journal di Chrome laptop tidak muncul di Safari HP │
  ├─────────────────┼────────────────────────────────────────────────────┤
  │ Browser-bound   │ Pindah browser = data tidak ikut                   │
  ├─────────────────┼────────────────────────────────────────────────────┤
  │ ~5-10MB limit   │ Cukup untuk ratusan entry, tapi ada batasnya       │
  ├─────────────────┼────────────────────────────────────────────────────┤
  │ Bisa terhapus   │ Clear browser data = hilang semua                  │
  ├─────────────────┼────────────────────────────────────────────────────┤
  │ Tidak bisa sync │ Tidak ada cloud backup otomatis                    │
  └─────────────────┴────────────────────────────────────────────────────┘

  Mitigasi yang sudah dibuat

  Fitur Export/Import JSON di halaman Journal dan Todos — backup manual ke file JSON, lalu restore di device/browser lain.

  Kalau mau lebih robust di masa depan

  Opsi upgrade kalau suatu saat dibutuhkan:

  1. Firebase/Supabase — free tier database + auth, bisa dipanggil langsung dari JS (tanpa backend). Data sync across devices.
  2. GitHub Issues sebagai storage — pakai GitHub API untuk simpan journal sebagai issue (hacky tapi gratis)
  3. Migrate ke Next.js/Vercel — keluar dari GitHub Pages, punya API routes, bisa konek ke PostgreSQL yang sudah ada

  Untuk saat ini, localStorage cukup memadai selama kamu satu device dan rajin export backup. Ini trade-off yang wajar untuk static site.
